# User-Agent 기반 헬스체크 필터링 개발 계획 (검증 완료)

## Context
현재 eBPF L7 HTTP 수집기는 경로(path) prefix 기반으로만 헬스체크를 필터링한다. ALB/NLB 헬스체크는 `/` 경로로 요청하기 때문에 기존 필터에 걸리지 않고 `internal_http_requests_total` 메트릭에 포함된다. 이로 인해 KEDA Zero Scaling 환경에서 워크로드가 영구적으로 scale-down 불가 상태가 된다.

**해결:** eBPF에서 User-Agent 헤더를 추출하고, Go 핸들러에서 User-Agent 패턴 매칭을 추가하여 ALB/kube-probe 헬스체크를 메트릭에서 제외한다.

---

## 리서치 검증 결과 (원래 계획 대비 변경점)

### 변경 1: 버퍼 128B → 256B 확장 (필수)

| 헬스 체커 | User-Agent 시작 offset | 128B 버퍼 | 256B 버퍼 |
| :--- | :--- | :--- | :--- |
| kube-probe (GET /) | ~42B | 가능 | 가능 |
| kube-probe (GET /healthz) | ~48B | 가능 | 가능 |
| ALB (최소 헤더) | ~93B | 간신히 가능 | 가능 |
| ALB (X-Forwarded- 포함)* | ~153B+ | 불가 | 가능 |

ALB가 X-Forwarded-For, X-Forwarded-Proto, X-Forwarded-Port, X-Amzn-Trace-Id 헤더를 포함하면 User-Agent가 153B+ 이후에 위치. 128B 버퍼로는 ALB User-Agent 추출이 불안정.

*   **스택 영향:** ~220B → ~348B (512B의 68%) — 안전 범위 내.

### 변경 2: #pragma unroll → bounded loop
`#pragma unroll`로 240회 반복을 unroll하면 ~3,600 인스트럭션이 생성되어 구형 커널(4K 제한)에서 검증기 거부 가능. 기존 코드(`http_trace.c:131`)에서 이미 bounded loop 사용 중이므로 커널 5.3+ 확인됨. bounded loop가 올바른 선택.

### 변경 3: "Kubernetes-Health" 패턴 제거
Kubernetes 소스 코드 검증 결과 `Kubernetes-Health` User-Agent는 실존하지 않음. kubelet은 `kube-probe/<major>.<minor>` 형식만 사용 (K8s 1.8+ PR #47729).

### 검증 완료 항목 (변경 불필요)

| 항목 | 결과 |
| :--- | :--- |
| BPF 스택 (512B) | ~348B 사용 (68%) — `__always_inline`으로 caller 스택 공유 |
| 중첩 bounded loop | 240 x 31 = ~7,440 states — 1M 한계의 0.7% |
| 구조체 정렬 (C↔Go) | char[32] 추가 시 패딩 없음. 수동 오프셋 파싱으로 정렬 문제 회피 |
| 링 버퍼 정렬 | 136B는 8의 배수 — 패딩 낭비 없음 |
| bpf2go | 링 버퍼 이벤트 Go 타입 미생성 — 수동 파싱 유지 |
| 대소문자 | ALB/kube-probe 모두 표준 "User-Agent:" 사용 — case-sensitive 매칭 충분 |

## Step 2: `bpf/l7_receiver/http_trace.c` — User-Agent 파싱

### 2-1. `parse_http_request()` 시그니처 및 버퍼 변경

```c
static __always_inline void parse_http_request(
    const char *buf, __s64 len,
    char *method_out, char *path_out,
    char *ua_out)                    // ← 신규 파라미터
{
    char local_buf[256];             // 128 → 256 (ALB X-Forwarded-* 헤더 대응)
    int read_len = len > 255 ? 255 : len;

    if (bpf_probe_read_user(local_buf, read_len, buf) < 0) {
        method_out[0] = '\0';
        path_out[0] = '\0';
        ua_out[0] = '\0';           // ← 추가
        return;
    }
    local_buf[read_len] = '\0';

    // ... 기존 method/path 파싱 로직 동일 ...
```

### 2-2. User-Agent 추출 로직 (경로 파싱 완료 후 추가)

```c
    // ================================================================
    // User-Agent 헤더 추출
    // "User-Agent: " 패턴 검색 (12바이트: 'U','s','e','r','-','A','g','e','n','t',':',' ')
    // bounded loop — #pragma unroll 사용하지 않음 (커널 5.3+ bounded loop 활용)
    // ================================================================
    ua_out[0] = '\0';
    for (int s = 0; s < 240 && (s + 14) <= read_len; s++) {
        if (local_buf[s]   == 'U' && local_buf[s+1]  == 's' &&
            local_buf[s+2] == 'e' && local_buf[s+3]  == 'r' &&
            local_buf[s+4] == '-' && local_buf[s+5]  == 'A' &&
            local_buf[s+6] == 'g' && local_buf[s+7]  == 'e' &&
            local_buf[s+8] == 'n' && local_buf[s+9]  == 't' &&
            local_buf[s+10] == ':' && local_buf[s+11] == ' ')
        {
            int ua_start = s + 12;
            int ua_idx = 0;
            for (int k = ua_start; k < read_len && ua_idx < (MAX_USER_AGENT_LEN - 1); k++) {
                if (local_buf[k] == '\r' || local_buf[k] == '\n') break;
                ua_out[ua_idx++] = local_buf[k];
            }
            ua_out[ua_idx] = '\0';
            break;  // 첫 번째 매칭에서 종료
        }
    }
}
```

**BPF 검증기 안전성:**
*   외부 루프: 상한 240, 조건 `(s + 14) <= read_len`으로 범위 보장
*   내부 루프: 상한 31 (`MAX_USER_AGENT_LEN - 1`), `k < read_len`으로 범위 보장
*   최악 verifier states: ~240 x 15 + 240 x 31 x 10 ≈ 78,000 (1M의 7.8%)
*   스택: `local_buf[256]` + 지역변수 ≈ 300B (512B의 58%)

### 2-3. 호출부 수정 (sys_exit_read 함수 내)

```c

// 변경 후
```

```go
    patterns   []string // path 패턴 (기존)

    enabled    bool     // path 필터 활성화 (기존)
```

**기본 UA 패턴:**
```go
var defaultHealthCheckUAPatterns = []string{
    "elb-healthchecker",    // AWS ALB/NLB — "ELB-HealthChecker/2.0"
    "kube-probe",           // Kubernetes kubelet — "kube-probe/1.28"
}
```

**생성자 시그니처 변경:**
```go
func NewHealthCheckFilter(
    enabled bool, customPatterns string,
    uaEnabled bool, customUAPatterns string,
) *HealthCheckFilter
```

**UA 매칭 메서드:**
```go
func (f *HealthCheckFilter) IsHealthCheckUA(userAgent string) bool {
    if !f.uaEnabled || userAgent == "" {
        return false
    }
    uaLower := strings.ToLower(userAgent)
    for _, pattern := range f.uaPatterns {
        if strings.HasPrefix(uaLower, pattern) {
            return true
        }
    }
    return false
}

func (f *HealthCheckFilter) UAPatterns() []string {
    return f.uaPatterns
}
```

## 위험 요소 & 대응

| 위험 | 확률 | 영향 | 대응 |
| :--- | :--- | :--- | :--- |
| BPF 검증기 거부 | 낮음 | 빌드 실패 | bounded loop + 고정 상한으로 안전. |
| BPF 스택 초과 | 매우 낮음 | 빌드 실패 | ~348B / 512B (68%). |
| ALB 확장 헤더로 256B 초과 | 매우 낮음 | UA 빈 문자열 | 일반적 ALB 요청 ~188B. 초과해도 path 필터 fallback. |
| 구조체 크기 불일치 | 없음 | 파싱 오류 | 수동 오프셋 파싱, 테스트로 검증. |
