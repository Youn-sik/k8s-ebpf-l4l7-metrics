## 2. 근본 원인 (Root Cause)

### 결론 한 줄

`parseHTTPPayload`가 **HTTP 요청 라인/헤더가 정상 포맷이라고 무조건 신뢰**하고 파싱하기 때문에, eBPF가 잡아 올린 페이로드에 비-HTTP 바이너리나 CRLF가 없는 데이터가 섞이면 `method`/`path`/`comm` 라벨이 **유효하지 않은 UTF-8 바이트**가 된 채로 [`l7_handler.go:236`](../ebpf/cmd/agent/l7_handler.go#L236)의 `counter.WithLabelValues(...)`에 들어가 Prometheus가 panic을 던집니다.

### 결함 4가지 (겹쳐서 발현)

#### 결함 ① eBPF의 HTTP 판별이 4바이트 매직만 검사
[`http_trace.c:54-70`](../ebpf/bpf/l7_receiver/http_trace.c#L54-L70)의 `is_http_request`는 페이로드 첫 4바이트만 `GET `, `POST`, `HEAD`, `PATC`, `OPTI` 등의 매직과 비교합니다.

- 연결 재사용된 read이거나
- 첫 read에 헤더+gzip body가 같이 도착했거나
- 비-HTTP 바이너리 페이로드의 첫 4바이트가 우연히 매직과 일치(false positive)

하는 경우 eBPF는 "HTTP다"로 판정하고 256B를 그대로 ringbuf로 전달합니다.
→ **dv86h 케이스**(`POST\n\x12\xfe\xdb...`)는 첫 4바이트가 `POST`라 매직은 통과했지만 5번째 바이트가 공백(`0x20`)이 아니라 `\n`(`0x0A`) — 비-HTTP false positive의 결정적 증거.

#### 결함 ② CRLF 부재 시의 폴백이 잘못됨 (핵심)
[`l7_handler.go:261-270`](../ebpf/cmd/agent/l7_handler.go#L261-L270):
```go
firstLineEnd := bytes.Index(data, []byte("\r\n"))
if firstLineEnd < 0 {
    firstLineEnd = len(data)        // ← 페이로드 전체를 request line으로 간주
}
requestLine := string(data[:firstLineEnd])
parts := strings.SplitN(requestLine, " ", 3)
if len(parts) >= 2 {
    method = parts[0]
    path = applyPathLimit(parts[1])
}
```
`\r\n`을 못 찾으면 **256바이트 전체를 request line으로 취급**합니다. 그 안의 임의 바이너리/공백 위치에 따라 `parts[0]`(method 라벨), `parts[1]`(path 라벨)에 **임의 바이트 시퀀스가 그대로 박힙니다**.
→ **j682d**의 `path = /\x0e\xe5\x92\f/v1/*` 패턴이 이 경로입니다.

#### 결함 ③ User-Agent UTF-8 경계 무시 바이트 자르기
[`l7_handler.go:283-285`](../ebpf/cmd/agent/l7_handler.go#L283-L285):
```go
if len(userAgent) > 31 {
    userAgent = userAgent[:31]      // ← UTF-8 코드 포인트 한가운데서 잘림
}
```
UA 헤더 라인이 256B 안에서 안 끝나면 `lineEnd < 0`이 되어 `userAgent`에 **이어지는 헤더/바디 바이너리까지** 들어가고, `[:31]`을 하면 멀티바이트/바이너리 중간에서 잘려 invalid UTF-8이 됩니다.
→ **6875r**의 `TELB-HealthChecker/2.0"\x1d\x8b\x002...`가 이 케이스.

또한 `TELB-HealthChecker`는 헬스체크 UA 화이트리스트(`elb-healthchecker`, `kube-probe`)에 없어 필터링도 안 됩니다.

#### 결함 ④ Prometheus가 invalid UTF-8 라벨에 panic
`client_golang/prometheus/counter.go:284`의 `WithLabelValues`는 `utf8.ValidString` 실패 시 **즉시 panic**합니다. `processRecord`는 recover가 없어 panic이 goroutine을 타고 올라가 컨테이너가 죽고, kubelet이 재시작 → 같은 트래픽 패턴이 다시 도착 → 다시 panic. **6875r 94회, dv86h 80회, j682d 55회 재시작**은 이 루프 때문입니다.

### 라벨 오염 경로 매핑

| 라벨 | 소스 | 오염 경로 |
|---|---|---|
| `method` | `parts[0]` | CRLF 없을 때 request line 전체에서 첫 토큰 → 바이너리 가능 (dv86h) |
| `path` | `applyPathLimit(parts[1])` | 위 + split이 바이너리 안의 공백으로 잘림 (j682d) |
| `comm` | `bytesToString(event.Comm[:])` | 대체로 안전, null 종결 누락 시 garbage 가능 |
| `srcIP`/`ns`/`svc`/`pod` | `net.IP`, k8smapper | 안전 |

## 3. 환경적 격차 — 왜 고객 클러스터에서만 발생하는가

개발 클러스터에서는 동일 버전의 코드로도 재현되지 않습니다. 패닉이 찍힌 Pod 3개가 모두 고객 환경의 특정 서비스(`loki-distributor`, `mimir-distributor`, `mimir-gossip-ring`)에서 발생했다는 사실이 결정적 단서입니다.

### 요인 1. 관측성 스택의 대용량 바이너리 푸시 트래픽
- **loki-distributor**: Alloy/Promtail이 POST `/loki/api/v1/push`로 **gzip 또는 snappy 압축 protobuf** body 전송
- **mimir-distributor**: Prometheus remote_write가 POST `/api/v1/push`로 **snappy 압축 protobuf** 전송

정상 로그 직후 패닉이 난 이유: 이 요청들은 **요청 라인 직후 헤더 + `\r\n\r\n` + 바이너리 body가 한 read() syscall에 같이 도착**합니다. eBPF는 256B만 떠 가는데 그 안에 바이너리 body 일부가 같이 들어와 있고, userspace 파서가 헤더 끝을 못 찾아 결함 ②를 탑니다.

### 요인 2. 멀티테넌시/트레이싱 헤더로 인한 헤더 크기 폭증
운영 환경 트래픽은 다음 헤더들이 함께 붙어 옵니다:
- `Authorization: Bearer <긴 JWT>` (수백 B)
- `X-Scope-OrgID: <tenant>` (Mimir/Loki 멀티테넌시)
- `traceparent`, `tracestate` (W3C 트레이싱)
- `X-Request-ID`, `Content-Encoding`, `Content-Type`, `Content-Length`

결과: **헤더 전체가 256B를 쉽게 넘어갑니다**. 그러면 요청 라인의 `\r\n`은 찾더라도 **User-Agent 라인의 `\r\n`을 못 찾아** UA 값에 다음 헤더/바디 바이너리가 흘러 들어오고, `userAgent[:31]`로 자르면 멀티바이트/바이너리 한가운데에서 끊겨 invalid UTF-8이 됩니다. 개발 클러스터는 헤더 셋이 가벼워서 이 경로를 안 탑니다.

### 요인 3. mimir-gossip-ring — HTTP가 아닙니다
`mimir-gossip-ring`(dv86h Pod)은 HTTP가 아니라 Hashicorp **memberlist 프로토콜**(TCP 커스텀 바이너리)을 사용합니다. 결함 ①에 따라 memberlist 메시지 페이로드 중 첫 4바이트가 `POST`(0x504F5354)와 우연히 일치하면 eBPF가 HTTP로 오인하고 256B를 ringbuf로 올립니다. `POST` 뒤에 공백이 아니라 `\n`이 붙어 있는 것이 false positive 확정 증거. 개발 클러스터에는 Mimir gossip ring이 없어 이 false positive가 안 일어납니다.

### 요인 4. Tencent ELB 헬스체크 UA가 화이트리스트에 없음
[`l7_handler.go:106-108`](../ebpf/cmd/agent/l7_handler.go#L106-L108):
```go
var defaultHealthCheckUAPatterns = []string{
    "elb-healthchecker", "kube-probe",
}
```
고객은 **Tencent Cloud** 환경이라 LB가 `TELB-HealthChecker/2.0` UA로 헬스체크를 쏘는데 `"telb-"`는 화이트리스트에 없어 필터링 안 되고 카운팅 코드 경로로 들어옵니다. 개발 클러스터는 AWS ALB라 `ELB-HealthChecker`로 시작해 정상 필터링됩니다.

### 요인 5. 트래픽 볼륨
고객 클러스터의 관측성 push QPS가 개발 환경 대비 수십~수백 배 큽니다. 256B 절단/false positive처럼 **확률 1/10⁴~1/10⁵**로 터지는 엣지 케이스가 분당 한 번씩 재현되니까 컨테이너가 한 시간에 몇 번씩 재시작합니다. 개발 클러스터는 트래픽 볼륨이 작아 같은 버그가 있어도 며칠에 한 번도 안 터질 수 있습니다.

### 환경 요인 × 코드 결함 매트릭스

| 환경 요인 | 트리거되는 결함 | 관찰된 Pod |
|---|---|---|
| ① 바이너리 푸시 트래픽 | ②(CRLF 폴백) | 6875r, j682d |
| ② 거대 멀티테넌시 헤더 | ③(UA truncate) | 6875r |
| ③ memberlist 비-HTTP | ①(eBPF false positive) | dv86h |
| ④ TELB 헬스체크 | UA 화이트리스트 누락 | 6875r |
| ⑤ 트래픽 볼륨 | ④(recover 부재) | 전부 |

## 4. 수정 사항 (v0.0.21)

### 결함 ① method 화이트리스트 검증 (SKP-2067)
- `validHTTPMethods` 화이트리스트 추가 (GET/POST/PUT/DELETE/HEAD/PATCH/OPTIONS)
- 화이트리스트에 없는 method는 즉시 reject

### 결함 ② CRLF 폴백 제거 + 3토큰 검증 (SKP-2068)
- CRLF 미발견 시 `return ""` (256B 전체 request line 취급 제거)
- `len(parts) < 3` 검증 (HTTP version 필수)
- path UTF-8 검증 추가

### 결함 ④ defer recover() 추가 (SKP-2071)
- `processRecord`에 `defer recover()` 추가
- panic 시 해당 이벤트만 손실, Pod은 살아남음

### 결함 ③ UA truncate (미적용, 추후 조치)
- 현재 UA는 `WithLabelValues`에 전달되지 않아 panic 미유발
- 결함 ①②④ 수정 후 안정 상태에서 재현 및 조치 예정
