# L7 Handler Panic 조사 보고서

- **작성일**: 2026-05-11
- **대상 모듈**: `installer/k8s-ebpf-l4l7-metrics`
- **발생 클러스터**: pet-observability-cluster
- **현상**: `k8s-ebpf-l4l7-metrics` DaemonSet Pod이 고객 클러스터에서 분당 단위로 재시작 (Pod별 55~94회)
- **panic 위치**: [`l7_handler.go:236`](../ebpf/cmd/agent/l7_handler.go#L236) — `counter.WithLabelValues(...)` 호출 시 Prometheus가 invalid UTF-8 라벨에 panic

## 2. 근본 원인 (Root Cause)

`parseHTTPPayload`가 수신 데이터를 정상 HTTP 포맷이라고 무조건 신뢰하고 파싱하기 때문에, 비-HTTP 바이너리나 비정상 형식의 데이터가 method/path 라벨에 유입되어 Prometheus가 panic합니다.

### 결함 4가지 (겹쳐서 발현)

#### 결함 ① eBPF의 HTTP 판별이 4바이트 매직만 검사
[`http_trace.c:54-70`](../ebpf/bpf/l7_receiver/http_trace.c#L54-L70)의 `is_http_request`는 페이로드 첫 4바이트만 HTTP method 매직과 비교합니다. 비-HTTP 바이너리의 첫 4바이트가 우연히 `POST`(0x54534F50) 등과 일치하면 HTTP로 오인하여 256B를 ringbuf로 전달합니다.

고객 로그 근거: dv86h의 `POST\n\x12\xfe\xdb...`는 첫 4바이트가 `POST` 매직과 일치했지만, 5번째 바이트가 공백(0x20)이 아닌 `\n`(0x0A)으로, 정상 HTTP가 아닌 바이너리 데이터입니다.

#### 결함 ② CRLF 부재 시의 폴백이 잘못됨
[`l7_handler.go:261-270`](../ebpf/cmd/agent/l7_handler.go#L261-L270):
```go
firstLineEnd := bytes.Index(data, []byte("\r\n"))
if firstLineEnd < 0 {
    firstLineEnd = len(data)        // ← 256B 전체를 request line으로 취급
}
```
`\r\n`을 못 찾으면 256바이트 전체를 request line으로 취급하여, 바이너리 데이터가 method/path 라벨에 유입됩니다.

고객 로그 근거: j682d의 `path = /\x0e\xe5\x92\f/v1/*` — 바이너리 중 우연한 공백(0x20) 위치에서 split되어 path에 바이너리가 유입된 패턴.

#### 결함 ③ User-Agent 바이트 단위 절단
[`l7_handler.go:283-285`](../ebpf/cmd/agent/l7_handler.go#L283-L285):
```go
if len(userAgent) > 31 {
    userAgent = userAgent[:31]      // ← 바이트 단위 절단
}
```
31바이트에서 바이트 단위로 절단하므로 멀티바이트 UTF-8 시퀀스 중간에서 잘려 invalid UTF-8이 됩니다.

고객 로그 근거: 6875r의 `TELB-HealthChecker/2.0"\x1d\x8b\x00...` — UA에 바이너리가 혼입된 후 절단.

추가로, 고객 로그에서 확인된 `TELB-HealthChecker`는 기존 헬스체크 UA 화이트리스트(`elb-healthchecker`, `kube-probe`)에 없어 필터링되지 않고 카운팅 경로로 진입합니다.

#### 결함 ④ processRecord에 recover 없음
`processRecord`에 `defer recover()`가 없어, 결함 ①~③에서 발생한 panic이 goroutine을 죽이고 컨테이너 재시작 루프를 유발합니다. 고객 로그의 재시작 횟수(94회, 80회, 55회)가 이 루프 때문입니다.

## 3. 환경적 격차 — 왜 고객 클러스터에서만 발생하는가

개발 클러스터에서는 동일 코드로 재현되지 않습니다. 고객 로그에서 확인된 서비스(`loki-distributor`, `mimir-distributor`, `mimir-gossip-ring`)의 트래픽 패턴이 결함을 트리거합니다.

### 요인 1. 바이너리 body를 포함한 HTTP 트래픽
고객 로그에서 확인된 loki-distributor, mimir-distributor는 압축된 바이너리 body를 POST로 수신하는 서비스입니다. eBPF가 캡처하는 256B 안에 HTTP 헤더와 바이너리 body가 함께 포함될 수 있으며, 파서가 이를 올바르게 구분하지 못합니다.

### 요인 2. 비-HTTP 바이너리 프로토콜
고객 로그에서 확인된 mimir-gossip-ring(dv86h)의 경우, 서비스 이름과 panic 데이터(`POST\n\x12\xfe\xdb...`)로 볼 때 비-HTTP 바이너리 통신이 eBPF에 의해 HTTP로 오인된 것으로 판단됩니다.

### 요인 3. 헬스체크 UA 미필터링
고객 로그에서 확인된 `TELB-HealthChecker/2.0`은 기존 화이트리스트에 없어 필터링 없이 카운팅 경로에 진입합니다.

### 요인 4. 트래픽 볼륨
고객 클러스터의 트래픽 볼륨이 개발 환경보다 훨씬 크기 때문에, 확률적으로 낮은 엣지 케이스가 분당 단위로 발현됩니다.
